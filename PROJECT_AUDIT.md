# Аудит проекта Sushi de Maksim (07.03.2026)

## 🏗 Общее состояние базы

- **Сборка (Build)**: Проект компилируется с помощью Vite 4.5.14 за `~1.63s` без критических ошибок.
- **Линтинг (Linting)**: Код в целом чист, проведено форматирование (устранено ~390 мелких предупреждений о стилистике). Имеется несколько предупреждений `eslint`, касающихся неиспользуемых переменных (`supabase`, `sql`) и пропущенных зависимостей в `useEffect`.
- **Типизация (TypeScript)**: Выполнен проход `tsc --noEmit`. Существуют места с использованием `any` (например, маппинг категорий в каталоге), которые в будущем требуют доработки для повышения безопасности.

## 🚀 Основные направления для улучшения (Roadmap)

### 1. 🛡️ Тестирование критических узлов (Automated QA)

В проекте сейчас полностью отсутствуют автоматизированные тесты. Так как это e-commerce (где ошибки напрямую означают потерю денег и заказов), добавление тестов должно стать важной задачей.

- **Unit-тестирование (Vitest)**: Написание тестов для логики расчётов (калькуляция корзины `finalTotal`, применение процентов `promoCodes`, проверка порога бесплатной доставки).
- **E2E-тестирование (Playwright/Cypress)**: Регулярная (раз в день/при деплое) автоматическая проверка пути пользователя: "добавление роллов -> ввод адреса -> оформление заказа".

### 2. 📡 Кэширование данных на клиенте (State Management)

Запросы к API построены через стандартные связки `useState / useEffect`. При росте приложения интерфейс будет загружаться дольше (на мобильных сетях).

- **Внедрение `@tanstack/react-query`**:
    - Меню и данные корзины будут кешироваться.
    - Автоматический умный polling состояния заказов (снимет нагрузку с ручных таймеров).

### 3. 🚦 Мониторинг ошибок в реальном времени (Error Tracking)

Сейчас настроен `@vercel/speed-insights` для метрик скорости, но нет трекинга "тихих" фронтенд/бэкенд ошибок.

- **Интеграция Sentry**: Установка агента для автоматического сбора отчетов (устройство, браузер, строчка кода), если у пользователя ломается интерфейс или не проходит оплата.

### 4. 🧹 Строгая типизация TypeScript (Type Hardening)

- Избавление от заглушек `any`.
- Создание строгих интерфейсов моделей БД для клиента и сервера, включение строгого режима в `tsconfig.json`.

### 5. 📧 Транзакционные Email-шаблоны (React Email)

Вместо отправки простых текстовых писем (через `nodemailer`) на случай подтверждения заказов или рассылок.

- **Внедрение `@react-email/components`**: Создание стилизованных, адаптивных HTML-писем с красивой таблицей товаров на базе существующих React-навыков.
